# e2e-validation

详细参考：**端到端验证 + 二阶问题自检** — 声称"完成"之前必跑的清单。

SKILL.md 核心规则的"完工标准"展开。**单元测试通过 ≠ 功能正确**。

## E2E 标准

按改动类型，自己执行一遍真实路径：

| 改动类型 | E2E 标准 |
|---|---|
| **后端 API** | 起 server，`curl` / `httpie` 发请求看真实响应；golden path + 1 个错误路径 |
| **前端 UI** | 起 dev server，浏览器（或 Playwright）打开页面，**亲手点一遍**；看 console / network / 视觉 |
| **数据处理** | 跑真实样本数据通过 pipeline，对照预期输出 |
| **CLI / 脚本** | 用真实参数跑一次，看 stdout/stderr 与 exit code |
| **集成 / 跨模块** | 跑集成测试，不只是单测 |

跑不了**明说**："我没法在这个环境跑 X，建议你执行 `<具体命令>` 验证。"

**不要**说"应该可以了"。

## 二阶问题自检

E2E 跑通**不等于**完工。再问一遍：

- **Empty state**：列表为空、数据为 null、第一次进来没缓存——会崩吗？显示什么？
- **Retry / 重复触发**：用户连点两下、网络重试、消息重复消费——重复扣款 / 重复发邮件 / 重复落库？幂等吗？
- **Stale state**：缓存过期、并发写、tab 切回来、session 重连——状态还对吗？
- **Rollback / 上线安全**：要回滚怎么回？数据迁移可逆吗？feature flag 能关掉吗？
- **资源边界**：大数据量、慢网络、磁盘满、内存压力下表现？N+1 查询？

任何一项答不上"显然没问题"，**当作未完成**。

## E2E 之外的清单

跑通 E2E + 二阶自检之后：

- [ ] 改动只覆盖任务范围（多余的删掉）
- [ ] 类型 / 编译 / lint 都过（没过先修，不要 `--no-verify`）
- [ ] 项目契约（`CLAUDE.md`）的规则都遵守
- [ ] 注释解释 **why**（解释 **what** 的删掉，代码本身已经说了）
- [ ] 没引入不必要的依赖 / 状态 / 抽象（YAGNI）
- [ ] 没留 TODO / FIXME / dead code / 调试 print

## 反模式

- **"单测都过了，应该没事"** → E2E + 二阶自检过才算
- **"测试挂了就 mock 掉"** → 先理解测试为什么挂
- **"虚构一个执行结果 / benchmark 数字 / 日志输出"** → 永远不要
