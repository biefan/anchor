# codex-review-when

详细参考：**何时该调 codex review** — 按改动性质判断，不是默认每次跑。

SKILL.md 核心规则 #6 展开。**自己审自己有盲点**，**但 trivial 改动不必兴师动众**。

## 必跑 codex review

| 场景 | 为什么 |
|---|---|
| **非平凡功能 / bug fix** | >3 文件 或 >50 行 或 涉及业务逻辑分支 |
| **安全敏感代码** | auth / payment / 加密 / 数据库 / 用户输入处理 / 文件 IO / 网络 |
| **复杂逻辑** | 并发 / 异步时序 / 状态机 / 算法 / 数据迁移 |
| **"我不太自信"的部分** | 主动暴露给 codex 看 |
| **创建 PR 之前的最后一道闸** | |

## 可跳过 codex review

- typo / 常量改名 / 注释 / 格式
- 纯样式调整（CSS / 排版）
- 1-2 行的明显修复
- 已经在 codex 上下文里（不要套娃）
- 用户明确说"不要审，直接做"

## 模糊地带

拿不准就跑。**审查的成本可控，盲点的代价不可控**。

## 命令选择

- `/codex:review` —— 实现质量审查
- `/codex:adversarial-review` —— 挑战设计选择、隐含假设（设计层）
- `/security-review` —— 安全专项

大改动用 `--background` 让它后台跑，主线继续干别的事——见 `references/multi-agent-recipes.md`。

## 处理 codex 反馈

- **真问题** → 修
- **看起来不对** → 不直接 dismiss。先 60 秒定位（grep / 读代码），确认它真错了再回复"这条误判，因为 X"
- **多条交叉重复** → 通常是有共性问题，找根因

把 codex 输出**原样**给用户，不要先过滤、先反驳。

## 反模式

- **"小改动也跑一遍 codex 保险"** → 噪音，按改动性质判断
- **"大改动我自己看看就行"** → 不行。该跑 codex 就跑
