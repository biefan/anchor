# intent-and-recon

详细参考：**意图清晰** + **60 秒定位** — 从用户原话到第一行代码之间该走的两步。

SKILL.md 核心规则 #1 + 写代码之前的"侦察"动作。

## 第一步：意图清晰才动手

含糊的需求 = 必然的返工。任何一项不清楚，**先问**：

- **范围**："改一下登录" → 哪个登录？密码 / OAuth / SSO？改行为还是 UI？
- **症状 vs 期望**："这里 bug" → 重现步骤？预期是什么？观察到什么？
- **关键决策**："加搜索" → 后端搜索 vs 前端过滤？模糊 vs 精确？分页？
- **影响面**："优化性能" → 哪个指标？接受什么取舍？

问的时候：
- 一次问 1-3 个**最关键**问题，不要一长串琐碎的
- 给具体选项让用户选（Claude Code: `AskUserQuestion`；Codex: 编号 1/2/3/4 让用户回数字）
- 标默认推荐 `(Recommended)`，便于用户快速 yes

什么时候不用问：已经具体到文件 / 函数 / 行号，或明显 trivial（typo / 常量）。

## 第二步：60 秒定位

意图清楚 + 项目契约已读（SessionStart 已注入）之后，**写第一行代码前**：

1. **找调用方与被调用方**。改函数前 grep 引用点；改接口前看上下游。
2. **找现有抽象**。要加 util？先看 `utils/` / `helpers/` / `lib/`。要加配置？先看现有 config 入口。
3. **找现有依赖**。要加包？先 grep manifest 看有没有等价的。
4. **说出假设**。猜数据结构 / 配置项 / 返回类型——停下来 grep / 读真实代码。
5. **一句话目标**。"让 X 从 A 变到 B"说清楚。说不清楚 = 还没想清楚。

搜索任务**能并行就并行**——见 `references/multi-agent-recipes.md`。

## 反模式

- **"用户没明说但我猜是这意思"** → 问
- **"这个 API 大概是这样调"** → 不要猜。grep 实际用例 / WebFetch 文档 / 看 SDK 源码
- **"全文件 Read 一遍保险点"** → 不保险，只是慢
- **"先返回一个半成品占位"** → 不要
